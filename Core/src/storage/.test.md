# Core::storage Sub-Module — Test Specification v3 (iter-015-core-storage)

Tests live in `Core/tests/test_storage*.rs` per Rust integration-test
convention. Sub-module-internal helpers live in `#[cfg(test)] mod tests`
inside `src/storage/*.rs` for private surface.

This file is the contract — every entry below maps to one `#[test] fn`;
every `#[test] fn` in code maps back here.

Because Storage has **no write methods**, tests build interesting states
either via:

- in-memory fresh storage (notes/history start empty)
- pre-built fixture SQLite files (notes/history seeded for read tests)
- pre-built fixture JSON files (dict seeded via `DictOrigin::SeededJson`)

## `test_storage_query_basic.rs`

Helper `mini_storage()` returns an in-memory `Storage` with one dict
`ee-builtin` seeded with `apple` / `apply` / `book` / `bookmark`, no
notes-fixture, no history-fixture.

- `query_empty_term_returns_immediately_finished_result` — `result.is_finished() == true`, `result.len() == 0`.
- `query_whitespace_term_treated_as_empty`.
- `query_trims_term_before_searching` — `"  apple  "` matches `apple` exactly.
- `query_returns_queryresult_immediately` — `Instant::now()` before/after `query` shows < 5 ms gap.
- `query_initial_is_finished_false_when_term_is_nonempty` — immediately after `query("app")`, `is_finished()` is `false` (worker may already be done in fast scenarios; tolerate that).
- `query_initial_was_cancelled_is_false`.
- `query_eventually_finishes_when_left_alone` — `wait_for(usize::MAX, Some(2s))`; assert `is_finished()`.

## `test_storage_query_growth.rs`

Each test waits via `wait_for` for deterministic results.

- `result_len_starts_at_zero_or_more_immediately_after_query`.
- `result_len_grows_monotonically` — sample `len()` repeatedly; values never decrease.
- `result_get_in_bounds_returns_some`.
- `result_get_out_of_bounds_returns_none`.
- `result_snapshot_reflects_current_records` — after `wait_for(3, _)`, `snapshot().len() >= 3` and entries match `get(0..3)`.
- `result_snapshot_is_independent_clone` — mutate the returned Vec; underlying list unaffected (subsequent snapshot still has original).
- `result_is_empty_iff_len_zero`.
- `result_wait_for_zero_returns_immediately`.
- `result_wait_for_count_returns_when_reached` — fixture has ≥ 5 candidates; `wait_for(5, Some(2s))` returns ≥ 5.
- `result_wait_for_unreachable_count_returns_on_finish` — fixture has 3 candidates; `wait_for(100, Some(2s))` returns 3 with `is_finished == true`.
- `result_wait_for_with_timeout_returns_after_timeout_if_not_reached` — `wait_for(usize::MAX, Some(50ms))` returns within ~50ms.
- `result_wait_for_returns_immediately_on_cancel`.

## `test_storage_query_multi_dict.rs`

Helper `multi_storage()` opens with two in-memory dicts:
- `dict-a`: `apple` / `axiom`
- `dict-b`: `apple` / `banana`

- `query_yields_records_from_all_dicts` — `query("apple")` snapshot contains a `DictRecord` from `dict-a` AND one from `dict-b`.
- `query_records_dict_id_matches_source` — `record.as_dict().unwrap().dict_id == "dict-a"` for dict-a records.
- `query_priority_breaks_score_ties` — when records from different dicts tie on score, the dict listed first in config appears first in the list.
- `query_no_dedup_across_dicts` — both `dict-a` and `dict-b` have `apple`; result list contains both records (count == 2 for that headword).

## `test_storage_concurrency.rs`

- `arc_storage_shared_across_threads_compiles_and_queries_work`.
- `parallel_queries_dont_deadlock` — 8 threads, each fires a query and drains via `wait_for(usize::MAX, Some(2s))`.
- `cancel_from_main_while_worker_iterates_terminates_worker_promptly` — main holds `Arc<QueryResult>`, spawned thread loops on `snapshot`; main calls `cancel`; spawned thread sees `is_finished == true` within 100 ms.
- `drop_queryresult_in_one_thread_observed_in_arc_clone_in_another` — `Arc<QueryResult>` cloned across threads; drop in one; the clone sees `is_finished == true`.
- `concurrent_queries_dont_corrupt_each_others_state` — 2 threads each iterate their own query handle; record counts independent.
