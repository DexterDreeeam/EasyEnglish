# Core::storage Sub-Module тАФ Test Specification (iter-015-core-storage)

Tests for this sub-module live in `Core/tests/test_storage*.rs` (per Rust
integration-test convention). Sub-module-internal helper tests may live in
`#[cfg(test)] mod tests` inside `src/storage/*.rs` where the unit under test
is private and not worth exposing.

This file is the contract: every test below maps to one `#[test] fn`; every
`#[test] fn` in code maps back here.

## `test_storage_open.rs`

- `in_memory_starts_with_empty_tables` тАФ `dict_count == 0`, `notes_count == 0`, `history_count == 0`.
- `in_memory_reports_schema_version_one`.
- `in_memory_returns_none_for_path`.
- `create_or_open_fresh_creates_file` тАФ supply a non-existent path; file exists after the call; `path()` returns the given path.
- `create_or_open_fresh_seeds_entries_table` тАФ `dict_count > 0` after `create_or_open(non_existent, seed_json)`.
- `create_or_open_existing_does_not_reseed` тАФ call `create_or_open` twice on the same path; `dict_count` stable; `entries` rows unchanged byte-for-byte (verified via SELECT digest).
- `open_missing_file_returns_io_error`.
- `open_unreadable_file_returns_io_error` тАФ set permissions to deny read (POSIX) or hold an exclusive write handle (Windows).
- `open_corrupted_file_returns_sqlite_or_corruption` тАФ write garbage bytes; assert error variant is `Sqlite` or `Corruption`.
- `open_newer_schema_version_returns_migration_error` тАФ manually bump `schema_version` to `CURRENT + 1`, reopen, expect `Migration { from: CURRENT+1, to: CURRENT, .. }`.

## `test_storage_history.rs`

Uses `Storage::in_memory()`. Tests rely on `history_record(word, max)`
behavior, not on a clock тАФ the implementation's internal clock is a private
test seam.

- `history_count_starts_zero`.
- `history_record_single_appears_in_recent`.
- `history_record_max_zero_is_noop` тАФ `history_record("apple", 0)`; `history_count == 0`.
- `history_record_dedup_same_word_keeps_one_row` тАФ record `apple` twice тЖТ `history_count == 1`.
- `history_record_dedup_updates_recorded_at` тАФ record `apple`, then `banana`, then `apple` тЖТ `recent[0].word == "apple"`, `recent[1].word == "banana"`.
- `history_record_evicts_oldest_when_over_cap` тАФ cap = 2, record `a`, `b`, `c` тЖТ `recent == [c, b]`.
- `history_record_evicts_in_correct_order` тАФ cap = 3, record 5 distinct words тЖТ only the latest 3, ordered most-recent-first.
- `history_record_dedup_is_case_insensitive` тАФ record `"Apple"` then `"apple"` тЖТ one row, last-seen casing recorded.
- `history_record_empty_word_returns_invalid_input`.
- `history_recent_max_zero_returns_empty`.
- `history_recent_caps_to_count_when_max_exceeds`.
- `history_clear_empties_the_table`.
- `history_clear_on_empty_is_noop`.

## `test_storage_concurrency.rs`

Uses `Arc<Storage>` cloned across threads. All tests build with
`#[test]` only (no `tokio` / `async-std`).

- `arc_storage_clone_does_not_reopen_file` тАФ share `Arc<Storage>`, both clones return the same `path()`; only one connection underneath (verified indirectly: write from clone A, read from clone B in same process).
- `parallel_readers_dont_deadlock` тАФ spawn 8 threads each calling `dict_lookup` 100 times; finish in bounded time.
- `parallel_writes_are_serialised` тАФ spawn 8 threads each calling `notes_set` 100 times with distinct words; all 800 notes present at end.
- `interleaved_read_write_consistent` тАФ one writer thread sets notes A-Z; one reader thread polls `notes_count`; reader count is monotonic-non-decreasing.

## `test_storage_seed.rs`

Uses `Storage::in_memory_with_seed(path)` with curated fixture JSONs.

- `seed_valid_loads_all_entries` тАФ fixture `seed_3.json` тЖТ `dict_count == 3`.
- `seed_empty_array_loads_zero_entries` тАФ fixture `seed_empty.json` (`[]`) тЖТ `dict_count == 0`, no error.
- `seed_missing_file_returns_seed_read_error`.
- `seed_malformed_json_returns_seed_parse_error`.
- `seed_partial_corrupt_entry_returns_seed_parse_error` тАФ one valid + one missing-`headword` record; whole load fails; no entries inserted (tx rolled back).
- `seed_duplicate_headwords_in_input_dedup_to_last` тАФ input has `apple` twice with different defs; only one row remains, holding the second definition (UPSERT semantics).
